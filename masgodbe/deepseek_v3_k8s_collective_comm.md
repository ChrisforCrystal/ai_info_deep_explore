# DeepSeek V3 (671B MoE) 在 K8s 上的推理部署实战：集合通信原语的真实映射

你目前在做的 K8s 基础架构（Volcano + HAMI + LWS + 高性能网络插件）是为了让 AI 跑得又快又稳。为了让你更加具象化，我们直接拿目前最火爆的 **DeepSeek V3（或 R1）** 来做推演。

DeepSeek V3 是一个 671B 参数的超大 MoE（混合专家）模型。哪怕是最好的 8 卡 H200 (单机总显存 8\*141GB)，一台机器也绝对装不下它。**你必须使用多机多卡（比如 2 台甚至 4 台 8 卡机）来做分布式推理。**

这就意味着，你的 16 张或 32 张 GPU 会被分配到不同的 Pod 中，运行在不同的物理机上。此时，在模型加载和一次前向计算（生成一个 Token）的过程中，这些 GPU 之间**必须极其频繁地使用集合通信原语交换数据**。

我们来看看在这个真实场景里，Gather、Broadcast、All-Reduce、All-to-All 到底是怎么“动起来”的。

---

## 一、 阶段 1：Pod 启动与模型加载 (Initialization)

在你的 K8s 集群中，Volcano 调度了 4 个 Pod（每个 Pod 8 张卡，总计 32 张卡），并被 LWS 管理起来（1 个 Leader，3 个 Worker）。

### 1. `Broadcast` 的应用：控制面的命令同步

虽然模型权重极大，但在真正分发请求时请求体（输入的一个 prompt 比如“你好”）是很小的。

- **动作**：用户向 API Gateway 发送了请求，API 网关将请求路由到 Leader Pod (Rank 0)。
- **原语映射**：Rank 0 接收到请求后，必须通过 **Broadcast**，将这个请求（或转换后的 Prompt Token IDs）瞬间广播给其他所有的 31 个 Worker (Rank 1 - 31)。
- **意义**：保证所有的 GPU 节点都知道现在要处理什么内容，大家要在同一时间点开始同一轮推理（Step）。

### 2. 为什么不用 `Scatter` 加载模型？ (Sharded Checkpoint 机制的胜利)

- **过去的误区**：让 Rank 0 读取整个数百 GB 的模型，然后用 `Scatter` 把模型切成 32 块，分发给另外 31 张卡。这会把 Rank 0 的网卡撑爆，显存 OOM。
- **真实的加载**：现在使用 vLLM 或 SGLang 等推理框架，每个 Rank 启动时，**依靠你提供的 CSI 高并发存储**，各自直接去存储上并行 `Read` 自己需要的那一份切片（这种并行加载可以看作是打破了传统的单点 Gather/Scatter 瓶颈）。

---

## 二、 阶段 2：推理计算核心的通信 (Attention 层计算)

DeepSeek 中的多头注意力机制（Attention）如果跨多卡，通常使用 **张量并行（Tensor Parallelism, TP）**。

假设我们在同一台物理机内部（8 张卡组成一个 TP 组，走极其高速的 NVLink）：
输入数据在这 8 张卡上被分别乘上了一部分权重。每张卡都算出了**不完整的局部结果**。

### 1. `All-Reduce` 的疯狂调用

- **动作**：这 8 张卡必须将各自的“局部结果”加起来，并且每一张卡都需要这个完整的加和结果，才能继续执行下一层的计算。
- **原语映射**：这就是最经典的 **All-Reduce（聚合+全量分发）**，通常计算方式是求和（SUM）。由于模型有几十层，生成一个词就要经过所有的层。这意味着在极短的时间内（毫秒级），这 8 张卡要疯狂执行几十次 `All-Reduce`。
- **与你架构的关系**：
  在配置 CRI（提供拓扑）和 TKE 调度时，由于所有的 `All-Reduce` 通信带宽极高，你必须确保 Volcano 把这 8 个形成 TP 组的 Pod **死死锁在同一台物理硬件**上。一旦跨机器（必须走以太网 / InfiniBand），速度会慢上几十倍，整个集群的吞吐量就崩盘了。

---

## 三、 阶段 3：MoE 混合专家路由的终极挑战 (Expert Parallelism)

这是 DeepSeek 这类 MoE 模型最特殊、压力最大的地方！
DeepSeek V3 中有 256 个“专家”（可以理解为数百个微型神经网络），这些专家被打散分布在你的 4 台机器（32 张卡）上。这种切分叫做**专家并行 (Expert Parallelism, EP)**。

当模型在一层中算出当前这个 Token 应该交给“专家 15”和“专家 128”来处理时，灾难来了：这两位专家可能分别位于节点 1 的 GPU 3 和节点 4 的 GPU 7 上。

### 1. `All-to-All`：网络风暴降临

- **动作**：在这一层，每台机器上的 GPU 都在疯狂切分它手里的 Token 激活值数据，并向**所有其他机器**上的对应 GPU 发送数据（把 Token 扔给对应的专家）；同时，也要接收**所有其他机器**传来的 Token（自己的专家要处理别人的数据）。专家算完后，又要再执行一次全量重排发送回去。
- **原语映射**：这就是 **All-to-All** 集合通信。它是整个 AI 推理中**最恐怖的跨网操作**，是真正意义上的“所有点对所有点的全量交换”。
- **与你架构的关系**：
  - 这就解释了为什么机房里需要配置 RDMA（比如基于 RoCEv2 的高性能网络）。
  - 在你的 CNI 插件设计中，你需要**给容器直通支持 SR-IOV 的网卡 (VF)**，并且启动 **GPU Direct RDMA (GDR)**。
  - GDR 的作用是：当机器 A 的卡 0 要向机器 B 的卡 7 进行 `All-to-All` 数据传输时，数据直接从 A 卡 0 的显存 -> PCIe 总线 -> A 的物理网卡 -> 光纤 -> B 的物理网卡 -> PCIe -> B 卡 7 的显存。全程不需要宿主机 CPU 的内存（RAM）来当二传手。如果没有这个高速 CNI 加持，跑这种涉及 `All-to-All` 的 MoE 模型就是痴人说梦。

---

## 总结：在 K8s 上拉起一个前沿大模型推理的过程

一句话概述全流程的通信过程：

1. **部署发起**：你的 Volcano 寻找合适的物理拓扑位置（尽量减少跨核心交换机），利用 HAMI 锁定 32 张卡。LWS 原子性地把它们一起拉起来 (Gang Scheduling，避免部分节点假死）。CRI 透传了 PCIe 和 NUMA 拓扑。
2. **加载分片 (并行的 IO 读)**：大家通过你的高并发 CSI 读取各自的权重副本。
3. **接受请求 (Broadcast)**：Rank 0 收到用户请求，通过 `Broadcast` 广播给所有节点。
4. **同机计算 (All-Reduce)**：在单机的张量并行（TP）计算环节，8 张卡互相使用 NVLink 执行高频低延迟的 `All-Reduce`。
5. **跨机专家路由 (All-to-All)**：当进入 MoE 层时，32 张卡瞬间跨机执行 `All-to-All` 交换数据。此时你配置的 SR-IOV 容器网卡直通和 RDMA 网络发挥奇效，数据绕过了 OS Kernel 和 CPU，扛住了巨大的网络风暴。
6. **汇聚输出 (Gather 或只在 Rank 0 输出)**：当算出结果后，所有的特征或者概率回到最后阶段，往往通过小范围的 `Gather` 或者直接由最后一个 Pipeline 的 Rank 将生成好的“字”返回给 API Gateway，推送给用户。
7. **继续下一个 Token……**

只要上述任何一个环节（如调度未感知拓扑导致跨交换机 All-Reduce，或者网络没直通导致 CPU OOM），推理速度就会从每秒生成几十个 Token 掉到每秒几个词，集群算力直接报废。这就是大模型时代为什么一定要深度研发底层 Infra（调度+网络+存储）的核心原因！
