在NCCL完成集群函数操作（如allreduce/allgather等）时，底层拓扑逻辑是如何建立的？以及通信算法Ring/Tree/Collnet等如何选择、源码的运行逻辑是怎么样的？本文围绕这些问题简要分析。

NCCL的大致工作流如下，首先进行通信topo的探测，比如有些什么样通信设备/硬件以及各个路径的参数；接着进行通信图的搜寻，完成ring图/tree图/collenet图的建立、对各个算法的时间消耗进行预估；最后通信的kernel执行，涉及一些CUDA操作。

topo的建立与微调优化（tuning）预测发生在Graph search过程，本文代码分析基于NCCL版本2.22，不同版本逻辑存在差异。

拓扑图的建立
当前NCCL中topo图主要有以下几个类型：

#define NCCL_TOPO_PATTERN_BALANCED_TREE 1 // Spread NIC traffic between two GPUs (Tree parent + one child on first GPU, second child on second GPU)
#define NCCL_TOPO_PATTERN_SPLIT_TREE 2 // Spread NIC traffic between two GPUs (Tree parent on first GPU, tree children on the second GPU)
#define NCCL_TOPO_PATTERN_TREE 3 // All NIC traffic going to/from the same GPU
#define NCCL_TOPO_PATTERN_RING 4 // Ring
#define NCCL_TOPO_PATTERN_NVLS 5 // NVLS+SHARP and NVLS+Tree
#define NCCL_TOPO_PATTERN_COLLNET_DIRECT 6 // Collnet Direct
NCCL最先支持的是ring形状的topo，后面随着集群规模扩大，推出了双向带宽更优的tree算法（参看：NCCL 2.4 Blog）以及直接利用网络进行allreduce的 Collnet Direct，后面推出的 NVLS+SHARP。topo图的建立过程主要完成：根据集群的硬件设备信息来查找/计算每个topo算法下的带宽与时延。该带宽与时延是预估值，为后面的算法选择提供输入。

算法的选择逻辑
算法的选择是指根据具体通信的数据来选择一个最优算法。首先进行算法带宽与时延的计算，然后根据通信的数据量计算时间消耗，并选择最小时间消耗的通路作为最终执行kernel的通路。

NCCL中耗时计算公式的定义如下(nccl/src/graph/http://tuning.cc L441)：

_time = lat _ latCount + nBytes / (1000 \* bw);
其中：lat：基本时延 ；bw：算法带宽； nBytes：本次通信的数据量大小；latCount为时延倍数，ring和tree的计算不一样，定义为：

int latCount = algorithm == NCCL_ALGO_RING ? numPipeOps : DIVUP(numPipeOps, NCCL_MAX_DEV_WORK_BATCH_COLLS);
numPipeOps:是指在group管道中集群操作的数量。

拓扑通路算法带宽/时延影响因素较多，包括操作函数、路径类型、topo结构、设备架构、通信协议、通信协议。带宽/时延的计算主要是在：nccl/src/graph/http://tuning.cc 中实现， 关键函数：

ncclTopoTuneModel ： 在初始化中是确定各个通路的带宽、时延。
ncclTopoGetAlgoTime： 根据数据量评估出最短时间消耗的通信路径。
这两个函数的调用时机分别是：comm初始化、集群通信操作。运行逻辑如下所示：

拓扑通路算法带宽计算与时延

带宽计算：先计算总线带宽，然后计算算法带宽。 总线带宽在topo构建中已完成，信息存储在ncclTopoGraph中。总线带宽不能直接用于计算耗时，需要根据具体的应用场景来进行微调，所以就有了场景化的算法带宽。场景主要根据以下要素进行区分：

集群通信操作：AllReduce、AllGather、ReduceScatter、Broadcast等
通信算法：Ring/Tree/Collnet/NVLS/NVLSTree
协议：数据构建的协议影响速度，可选的protocol主要是三种，低延时/128B低延时/常规 对应参数：LL/LL128/'Simple。
算法带宽的计算过程：取算法基数值 busBw= ncclTopoGraph->bwIntra ，经过NCCL_ALGO/NCCL_PROTO/NCCL_TOPO等场景修正（即乘以一定的比例系数）后，把结果存储在comm中：

参数含义：coll: 集群通信操作；a: 通信算法；p: 协议
时延计算：首先考虑基础时延：comm->latencies[coll][a][p] = baseLat[a][p];

带宽与时延的初始化，不同通信算法与通信协议的基础时延定义：

// Latencies in us, Bandwidths in GB/s
// Tree { LL, LL128, Simple } , Ring { LL, LL128, Simple }
static const float baseLat [NCCL_NUM_ALGORITHMS][NCCL_NUM_PROTOCOLS] = {
{ 6.8, 14.0, 0 }, { 6.6, 14.0, 8.4 }, // Tree, Ring
{ 0, 0, 0 }, { 0, 0, 0 }, // Collnet Direct, Chain
{ 0, 0, 0 }, { 0, 0, 0 }}; // NVLS, NVLS Tree

hwLat计算获得interlat，然后用interlat经过修正后叠加到comm->latencies中。 不同的算法需要不同的计算公式：

树形：comm->latencies[coll][a][p] += 2 _ ((nRanks/nNodes-1) _ intraLat + log2i(nNodes) _ interLat);
COLLNET_DIRECT：comm->latencies[coll][a][p] += 2 _ (std::min(1, (nRanks/nNodes-1)) _ intraLat + (nRanks/nNodes-1) _ 0.4) + interLat;
...
这样刷新了算法的时延comm->latencies

算法耗时预估与选择
算法的选择过程发生在NCCL集群通信函数调用中：先用ncclTopoGetAlgoTime计算通信时间，后选择topo图里面的时间最短的路径。大致步骤：

调用集群操作；
集群操作在group中入队，然后触发launch；
队列准备过程中完成算法选择；
算法选择用ncclTopoGetAlgoTime进行耗时评估；
选择最小时间消耗作为本次通信操作的算法，等待执行。
