# 为什么 671B 参数的大模型，张量并行 (TP) 能放进一台物理机？

这是一个直到今天依然困扰无数工程师的“终极困惑”！

你刚才潜意识里有一个非常致命的逻辑推导：

> “大模型有 6710 亿 (671B) 参数 -> 显存要 700GB -> 既然张量并行 (TP) 是切分模型，那为了把这 671B 塞下，我的 TP 就必须跨机器切开啊！怎么可能限制在一台机子里呢？”

这个逻辑对于传统的**稠密模型 (Dense Model)**（比如 LLaMA-3 400B、GPT-3 175B）是完全正确的。如果是稠密模型，TP 确实装不下，必须用 流水线并行 (PP) 或者全参数分片 (FSDP / Tensor Parallelism) 强行跨机。

**但 DeepSeek V3 是 MoE（混合专家模型）！** 在 MoE 里，参数的分布是极其不均匀的！

这正是我们要拆解的核心奥秘：

---

## 核心奥秘：大头不在 TP，而是在专家 (EP) 手里！

让我们直接解剖 DeepSeek V3（671B 参数）的身体结构。它的 671B 并不是铁板一块，而是分为极其悬殊的两个部分：

### 1. 稠密部分（Dense Parameters）：真正的 TP 参与者

这部分包含了：词嵌入层 (Embedding)、多头注意力机制 (Attention 的 Q, K, V, O 投影矩阵)、所有的 LayerNorm 等。

- **这部分的参数量是多少呢？**
  只有约 **200~300 亿 (20B ~ 30B) 参数！**
- **显存占用：** 若用 FP8 或 BF16 精度，这部分参数只需要区区 **30 GB 到 60 GB** 的显存！
- **K8s 下的归宿：同一台机器。**
  这微小的几十 GB，放在一台拥有 8\*141GB (1.1TB) 的 H200 机器上简直是九牛一毛。所以，我们使用 **张量并行 (TP=8)**，只把这 30B 的公共部分切成了 8 份，放在同一台机器的 8 张卡上。
  由于就在同机，它们跑极高频的 `All-Reduce` 非常快。

### 2. 专家部分（Expert Parameters）：必须跨机的“虚胖”体重

真正的体重全在这里！DeepSeek V3 每一层都有多达 256 个专家（其实就是替换了原来的 FFN 层的小型全连接网络）。

- **这部分的参数量是多少呢？**
  高达 **6400 亿 (640B)！** 几乎占了 95% 以上的总体积。
- **显存占用：** 需要 **将近 700 GB 到 1 TB** 显存才能勉强塞下。
- **K8s 下的归宿：全员打散（跨机 EP）。**
  既然 640B 太大了，那我们就在 4 台机器（32 张卡）上，**每张卡分配 256 / 32 = 8 个专家**。这就是所谓的 专家并行 (Expert Parallelism, EP)。

---

## 一次前向推理：TP 与 EP 在 K8s 上的交响乐

我们来看看在这个模型架构下，当你在 K8s 集群里生成一个字，到底发生了什么美妙的接力赛。

假设你用 4 台机器（Node 1 ~ Node 4），总共 32 张卡部署了 DeepSeek V3。

#### 第一阶段：所有机器都在跑 TP（多副本）

每一台机器（Node 1、Node 2、Node 3、Node 4）内部，**都有一份一模一样的、完整的上述“稠密部分（30B）”**！

- 此时，Token 进来了。Node 1 上的 8 张卡，用极快的 NVLink 执行 TP，在一瞬间完成了 Attention 层的多头注意力计算。
- **注意：这 30B 的代码逻辑和权重，这 4 台机器上是复制的（你可以理解为在这 30B 稠密参数上，它们是 数据并行 DP 的关系）。** 他们互不干涉地在自己机器内部算出了自己负责的 Token 的 Attention 特征。

#### 第二阶段：Token 走到门口，遭遇 EP（跨机风暴）

当 Attention 层算完，Token 来到了这一层的**重头戏：混合专家门控网路 (MoE Router)**。

- Token 敲门问路由器：“我该去找哪几个专家？”
- 路由器一看，对 Node 1 说：“你的这个 Token，最适合的专家是【专家 15】，他在 Node 2 的 3 号卡上；第二个适合的专家是【专家 200】，他在 Node 4 的 7 号卡上。”
- **跨机 All-to-All 爆发**：立刻，Node 1 的这 8 张卡，通过你给配置的 **直通网卡 (RDMA)**，把这些 Token 像发快递一样，跨过机柜交换机，直接砸到了 Node 2 和 Node 4 的对应显存里！
- Node 2 和 Node 4 的卡接收到这些“外包”代码（Token 特征），用自己庞大的专家权重进行计算。
- 计算完，再次原路跨网络发回去（又是一次 All-to-All）。

#### 第三阶段：收工回单机

特征数据拿回来后，各台机器继续在自己的单机内部，算下一个模块（或者是下一层的 Attention），进入下一轮的 TP 并行计算。

---

## 终极总结给你的 AI Infra 直觉

大模型并不是一个巨大的实心铁块，它是一个 **“公共部分很小（Attention），分支部分极大（Experts）”** 的树形结构！

1. **TP（实心打碎）：** 切分的是那 30B 的**公共基础结构（Attention）**。因为每一次计算都要全员高频沟通，所以必须死死锁在**同一台物理机（TP=8）** 内，利用 NVLink 互相传数据。
2. **EP（外包分包）：** 分离的是那 640B 的**专家仓库（MoE）**。因为太庞大，必须**跨机（EP=32 等）** 存储。反正不是每次每个 Token 都要用到所有专家，只要通过高速**跨机 RDMA** 发个快递去请专家算一下拿回结果就行。

这也就是为什么：你的 K8s 必须确保 TP 组必须同机调度，而 EP 组虽然跨机，但你必须为它们提供牛逼的直通网络（SR-IOV + GDR），因为它们互相丢包的频率，决定了“跨机外包”的速度！
