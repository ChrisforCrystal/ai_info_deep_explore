大模型的推理中，相同的输入可以产生的不同输出，但当我们需要让相同输入得到相同输出时，该如何实现？在Debug、强化学习、精度对齐这类场景中，我们有时会期望模型具备重复输出的能力，也就是可复现性。有开发者尝试关闭随机采样机制，却发现依旧无法得到完全一致的输出结果。这一问题的根源究竟是什么？本文将带读者快速了解其中奥秘，同时介绍vLLM、SGLang中启动确定性运算的方式。

大模型推理结果的不确定性可分为两类：一类是符合预期的、我们主动追求的不确定性，另一类则是超出预期的、非期望的不确定性。下面我们先对第一类展开分析。

1 非确定性(随机)采样
大模型的推理生成中，为了结果的多样性，通常会引入改变采样结果的操作，比如Temperature/Topk/TopP/MinP等（在“LLM推理采样(Sampling)”中有过详细介绍），但对于相同输入，大部分计算得到的输出是不变的。只有使用了随机因子的操作，才会出现输入相同而生成结果不同，比如权重采样。

随机采样的一般过程。
除了随机采样，还存在确定性采样方式，例如贪婪算法、Beam search。这类采样主要适用于机器翻译、概要生成等场景，这类场景无需输出多样性，核心需求是让结果更精准。

因此，若要使相同输入对应的采样结果保持一致，可采取以下两种方式：

固定非确定性采样的随机因子(即seed值)；
使用确定性采样方式，例如贪婪算法(设置temperature=0)。
尽管采用了确定性采样，且固定了所有随机因子（如pytorch的torch.manual_seed、numpy库的 np.random）但在推理框架中仍无法实现计算结果的可复现，尤其在处理大批量数据时，更准确地说，在数据批次动态变化时，这一问题会尤为突出。接下来我们就进入对第二类不确定性的分析。

2 计算的不确定性
在计算机系统中，浮点数运算并不严格遵循加法结合律，可能出现：

示例代码：

import numpy as np

# float32能精确表示的最大连续整数：2^24=16777216

a = np.float32(16777216.0)
b = np.float32(1.0)
c = np.float32(-16777216.0)

# 强制存储结果，避免硬件扩展精度干扰

left = np.float32((a + b) + c)
right = np.float32(a + (b + c))
print("是否相等:", left == right) # 输出 False
本质上是因为浮点数的计算顺序会影响结果。为什么数据的计算顺序会产生变化？

计算机中，算法库为了提升计算性能，会根据数据大小的不同来采用不同的算法策略。如在GPU设备中，会针对不同数据块大小，采用不同的tiling策略。

尽管单个算子引发的计算结果差异十分细微，但当这类差异逐步累积并达到一定程度时，就会对最终的输出结果产生显著影响。这也是推理最后的输出结果具有不确定性的关键原因。

2.1 Triton-Reduction示例
我们用Triton库构建一个演示示例，通过调整BLOCK_SIZE参数来对比不同情况下的计算差异 ，其中BLOCK_SIZE对应CUDA中线程(threads)的组织单位CTA[1]。

线程组织差异的示例
定义算子。使用tl.sum（默认树形算法）函数来进行规约计算，算子实现如下：

@triton.jit
def reduction_kernel_simple(
x_ptr, # 输入指针
output_ptr, # 输出指针
n_elements, # 元素数量
BLOCK_SIZE: tl.constexpr, # 线程块大小
):
"""简单的归约核函数"""
pid = tl.program_id(axis=0)
block_start = pid \* BLOCK_SIZE
offsets = block_start + tl.arange(0, BLOCK_SIZE)

    # 加载数据
    mask = offsets < n_elements
    x = tl.load(x_ptr + offsets, mask=mask, other=0.0)

    # 简单的归约
    accumulator = tl.sum(x)

    # 存储结果
    tl.store(output_ptr + pid, accumulator)

2.2 框架中的特性
在了解GPU运算相关知识后，我们再来探讨：为何在大模型推理场景下，GPU线程块会发生变化或出现算法切换？主要的原因在于动态批处理（dynamic batching）与多进程处理这两种机制。

动态的批处理。动态批处理的产生机制主要有两方面：一是各数据长度存在差异且会动态变化；二是与continuous-batch（连续批处理）的处理模式相关，该模式下一个请求处理完成后会立即下发新的请求。

动图封面
continuous batching
多进程处理。为了提升推理实例的处理性能，当前主流推理框架均支持多进程处理特性。以vLLM V1为例，即便下发的是相同数据，受调度机制的影响，每个engine core获取到的请求数据存在差异[3]。

vLLM框架中多engine cores协同工作
数据批次的变化会影响GPU计算算子的tiling块大小设定以及算子的算法选型，进而改变算子的运算执行顺序，最终造成推理结果无法复现。

3 推理框架的中的解决方案
我们知道，在pytorch中可通过设置torch.use_deterministic_algorithms (True) 来规避部分非确定性运算的产生，以此满足调试场景下的使用需求。而在推理框架的应用场景中，还存在另一类需求：RL（强化学习）要求Rollout输出确定的结果，以保障训练精度。目前，两大主流框架对确定性运算都有支持，下面依次介绍。

3.1 vLLM框架
在vLLM中先是通过设置参数来消除调度的不确定性[4]。

# 设置环境变量

V1 中设置参数：
VLLM_ENABLE_V1_MULTIPROCESSING=0
动态batch修改方式：

# 设置环境变量

VLLM_BATCH_INVARIANT=1
同时，设置全局种子seed（或者采用贪婪采样）：

sampling_params = SamplingParams(
temperature=0.7,
top_p=0.95,
max_tokens=100,
seed=42,
)
Batch Invariance打开后，vLLM计算会进行如下调整[5]：

1、对注意力机制及其他运算采用确定性内核实现方案
2、确保在不同批次大小下均能呈现一致的数值表现
3、禁用可能引入非确定性的特定优化手段（例如张量并行模式下的自定义全归约操作）
值得注意的是：这些调整对vLLM的整体性能会产生一定的负向影响。

当前Batch Invariance适配的模型有DSV3、Qwen3、Llama3等，但硬件需要GPU计算能力>=9.0（如Hopper/BlackWell架构）。

Batch Invariance处于“beta”状态，可以密切关注相关的issue：27433[6]。

3.2 SGLang框架
在确定性运算支持层面，SGLang框架已推出正式版本提供该能力支持。使用方式是：在推理服务启动阶段添加指定的参数enable-deterministic-inference[7]。

SGLang的确定性运算功能，支持为随机采样配置多个固定种子的使用方式。客户端侧的请求示例如下：相同的seed值可使生成内容保持一致，不同的seed值则会产生不同的输出结果，该使用方式主要适用于RL场景。
